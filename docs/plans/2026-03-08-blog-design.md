# 个人博客首版设计文档

日期：2026-03-08

## 1. 目标

当前项目采用首版最小可用方案，目标是尽快完成一个可上线的个人博客闭环，并为后续前后端分离保留演进空间。

当前确认的技术与范围如下：

- 语言：Python
- 后端框架：FastAPI
- 数据库：PostgreSQL
- 页面方案：Jinja2 服务端渲染
- 后台方案：自建简洁后台
- 第一阶段范围：极简上线版
  - 文章
  - 分类 / 标签
  - 后台登录
  - 发布 / 编辑

## 2. 设计原则

本阶段遵循以下原则：

1. 先完成最小闭环，不提前实现评论、搜索、统计等扩展功能。
2. 采用单体 SSR 架构，以降低首版复杂度与部署成本。
3. 代码结构上预留 API 化能力，避免未来前后端分离时推倒重来。
4. 数据库内部关联使用 id，对外访问 URL 使用 slug。
5. 后台只服务管理员，不引入多角色权限系统。

## 3. 架构设计

### 3.1 当前架构

首版采用单体应用架构：

- FastAPI 负责路由、业务逻辑、认证
- Jinja2 负责前台页面与后台页面渲染
- PostgreSQL 负责数据持久化
- 静态资源由应用直接提供

该方案适合博客首版，具备以下优点：

- 开发路径短
- SEO 友好
- 部署简单
- 后续可平滑演进为 API + 前端分离模式

### 3.2 后续演进路线

后续前后端分离时，采用渐进演进方式：

1. 当前先把业务逻辑沉淀在 service 层
2. 前期页面由 Jinja2 渲染
3. 后期逐步补充 `/api/...` 接口
4. 视需要把前台或后台替换成 Vue / React

因此本阶段不是临时方案，而是可演进方案。

## 4. 功能范围

### 4.1 前台功能

- 首页文章列表
- 文章详情页
- 分类页
- 标签页

### 4.2 后台功能

- 管理员登录
- 文章列表管理
- 新增文章
- 编辑文章

### 4.3 暂不纳入首版

- 搜索

- 草稿 / 定时发布
- 附件管理
- 访问统计

## 5. 数据模型设计

### 5.1 User

用途：管理员登录

建议字段：

- id
- username
- password_hash
- is_active
- created_at

### 5.2 Post

建议字段：

- id
- title
- slug
- summary（可选）
- content
- created_at
- updated_at
- published_at

说明：

- 首版可以只保留“已发布”文章，不强制引入草稿状态
- 若后续扩展草稿能力，可补充 status 字段

### 5.3 Category

建议字段：

- id
- name
- slug

### 5.4 Tag

建议字段：

- id
- name
- slug

### 5.5 关系

- 一篇文章属于一个分类
- 一篇文章可关联多个标签

## 6. slug 设计

slug 用作公开 URL 的稳定标识。

示例：

- `/posts/fastapi-blog-setup`
- `/categories/python`
- `/tags/fastapi`

设计规则：

- 小写
- 空格转 `-`
- 尽量只保留字母、数字、连字符
- 同类型下唯一

实现原则：

- 数据库内部关联使用 `id`
- 对外页面访问使用 `slug`

这样可以兼顾内部稳定性与外部可读性，也利于后续 SEO。

## 7. 认证与安全设计

首版后台仅支持管理员登录，不做复杂权限模型。

建议方案：

- 使用 session / cookie 登录
- 密码仅保存哈希值
- 后台路由统一加鉴权
- 表单提交做基础校验
- slug 做唯一约束
- 公开页面与后台路由分离

原因：

- 对博客后台而言，session 模式更自然
- 首版不需要 JWT 带来的额外复杂度
- 未来若前后端分离，再补 token 型认证即可

## 8. 项目结构建议

建议从一开始按可演进方式组织：

```text
app/
  main.py
  core/
  routes/
  models/
  schemas/
  services/
  templates/
  static/
```

建议职责：

- `app/main.py`：应用入口
- `app/core/`：配置、数据库、认证基础设施
- `app/routes/`：前台路由、后台路由、后续 API 路由
- `app/models/`：数据库模型
- `app/schemas/`：表单 / API 数据结构
- `app/services/`：文章、分类、标签、认证等业务逻辑
- `app/templates/`：Jinja2 模板
- `app/static/`：静态资源

关键约束：

- 模板只负责展示
- 业务逻辑不直接散落在路由中
- 为未来 API 化复用 service 层做准备

## 9. 验证标准

首版完成后应满足以下验证标准：

- 能登录后台
- 能新增文章
- 能编辑文章
- 首页能展示文章列表
- 文章详情页可访问
- 分类页可用
- 标签页可用
- PostgreSQL 数据可正常持久化

首版不强求完整自动化测试体系，但至少要保证：

- 关键页面与路由可访问
- 基础 CRUD 正常
- 页面闭环完整

## 10. 风险与控制

### 10.1 风险

- FastAPI 不自带 admin，需要自行实现后台页面
- 认证、模板组织、项目结构需要自行约束
- 若一开始路由和模板耦合过深，后续前后端分离成本会上升

### 10.2 控制措施

- 严格区分 routes / services / models
- 首版仅实现最小功能，不扩需求
- 在数据结构和业务层面优先做可演进设计，而不是提前做复杂前端方案

## 11. 结论

本项目首版正式采用以下方案：

- FastAPI + Jinja2 + PostgreSQL
- 自建简洁后台
- 使用 slug 作为公开 URL 标识
- 第一阶段只做极简上线版

这是一个兼顾当前开发效率与后续演进空间的方案。后续若需要前后端分离，应以补充 API、复用业务层为主要演进路径，而不是整体重写。
