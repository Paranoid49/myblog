# 接口与 Service 全覆盖测试设计

**日期：** 2026-03-08
**主题：** 针对每个 HTTP 接口与每个 service 公开函数建立系统化测试用例

## 背景与目标

当前项目经过多轮功能推进后，出现了“线上链路报错后再补测”的被动节奏。为避免继续出现同类问题，本次目标是建立一套覆盖明确、可验证、可持续维护的测试体系：

1. 每个 HTTP 接口都有测试用例；
2. 每个 service 公开函数都有测试用例；
3. 每个单元至少覆盖主路径与关键失败/边界路径；
4. 全量回归可稳定通过。

## 范围

### 纳入本轮

1. **HTTP 层**（`app/routes/*`）
   - `public`
   - `setup`
   - `admin_auth`
   - `admin_posts`
   - `health`

2. **Service 层公开函数**（`app/services/*`）
   - 已存在业务服务模块中的公开函数
   - 重点包括：初始化、自动建库、迁移、认证、文章构建与更新、分类标签查询等

### 不纳入本轮

- 仅用于内部实现细节且不影响可观察行为的私有 helper 机械覆盖
- UI 样式层面的断言
- 与本轮目标无关的大规模重构

## 方案比较

### 方案 A：端到端优先
- 特点：HTTP 端覆盖尽量做重，service 只补关键分支。
- 优点：最贴近用户行为。
- 缺点：调试成本高、维护压力大、定位慢。

### 方案 B：分层覆盖（选定）
- 特点：HTTP 与 service 分层覆盖；每个对象至少“主路径 + 关键失败/边界路径”。
- 优点：覆盖面与可维护性平衡最佳，定位问题快，适合当前节奏。
- 缺点：需要前置整理清单并持续维护映射。

### 方案 C：契约表驱动后自动生成骨架
- 特点：先做契约数据化，再批量生成测试骨架。
- 优点：长期最规整。
- 缺点：短期投入高，不适合当前“快速收口风险”的优先级。

## 选定方案

选用 **方案 B：分层覆盖**。

原因：当前主要问题是稳定性与漏测，不是缺少测试框架能力。分层覆盖可以在较短时间内把风险面压住，并且保持后续迭代成本可控。

## 测试设计

### HTTP 层设计

每个 endpoint 至少 2 条用例：

1. **主路径用例**
   - 断言状态码
   - 断言关键响应（重定向目标、关键文本、关键字段）

2. **失败/重定向用例**
   - 未初始化重定向 `/setup`
   - 未登录重定向 `/admin/login`
   - 参数非法（400/422）
   - 资源不存在（404）

### Service 层设计

每个公开函数至少 2 条用例：

1. **主路径用例**
   - 正常输入下输出正确或数据库状态变化正确

2. **边界/异常用例**
   - 重复初始化、冲突、权限错误、外部依赖异常包装
   - 必要时断言异常类型与不变量

## 组织方式

优先在现有测试文件中扩展，避免碎片化：

- `tests/test_public_pages.py`
- `tests/test_admin_auth.py`
- `tests/test_admin_posts.py`
- `tests/test_setup_service.py`
- `tests/test_database_bootstrap_service.py`
- `tests/test_post_service.py`
- 必要时新增最少量测试文件（仅当现有文件无法承载）

## 防漏机制

执行前先产出并维护两份映射清单：

1. `route endpoint -> 测试用例`
2. `service 公开函数 -> 测试用例`

开发过程中按映射逐项勾选，确保覆盖完成可证明。

## 执行策略

1. 先盘点全部 endpoint 与 service 公开函数
2. 优先补齐 setup/public/admin_auth 主链路
3. 再补 admin_posts 管理链路
4. 最后补齐其余 service 层函数

全过程遵循 TDD：先红（失败测试）→再绿（最小实现）→回归。

## 风险与应对

1. **风险：测试数量快速增长导致维护难度上升**
   - 应对：统一 fixture 与断言风格，避免重复 setup。

2. **风险：外部依赖导致测试不稳定**
   - 应对：对外部边界做 mock，核心行为保留真实路径。

3. **风险：一次改动跨越过大引入连锁回归**
   - 应对：按模块分批推进，每批都跑模块级回归。

## 验证标准

1. endpoint 清单全部有测试映射
2. service 公开函数清单全部有测试映射
3. 分层回归通过：
   - `tests/test_public_pages.py tests/test_admin_auth.py tests/test_admin_posts.py`
   - `tests/test_setup_service.py tests/test_database_bootstrap_service.py tests/test_post_service.py`
4. 全量回归通过：`python -m pytest -v`

## 预期结果

完成后，项目将从“出错后补测”转为“需求/改动即有对应测试防线”，显著降低初始化链路、登录链路、内容管理链路与 service 规则变更带来的回归风险。
