# 中文 slug 升级设计文档

日期：2026-03-08

## 1. 背景

当前项目的 slug 生成逻辑仅适用于英文内容，规则为保留 `a-z0-9-`。在中文标题为主的博客场景下，该规则会导致中文字符被全部清洗，最坏情况下生成空 slug，因此不适合当前项目方向。

项目已确认：博客内容以中文标题为主，因此需要尽早将 slug 升级为中文友好方案，避免后续文章创建、分类和标签功能建立在错误 URL 规则之上。

## 2. 目标

本次升级目标是引入一套适合中文博客的 slug 规则，满足以下要求：

- 英文保留英文
- 中文转拼音
- 数字保留
- 标点、空白、特殊符号统一作为分隔符
- 最终 slug 使用 `-` 连接
- slug 结果稳定、可读、可唯一化

本次只升级 slug 规则引擎，不扩展文章 CRUD、分类管理或标签管理功能。

## 3. 方案对比

### 方案 A：使用 pypinyin 转拼音（推荐）

规则：
- 英文转小写
- 中文转无声调拼音
- 数字保留
- 标点与空白统一转为分隔符
- 连续分隔符合并为一个 `-`

优点：
- 最适合中文博客 URL
- 可读性最好
- SEO 与人工识别体验更好
- 规则统一直观

缺点：
- 需要新增一个轻量依赖

### 方案 B：中文直接保留

优点：
- 无需引入拼音依赖
- 实现简单

缺点：
- URL 混合中文与英文
- 复制、编码和兼容性体验不如拼音 URL
- 不符合项目“中文转拼音”目标

### 方案 C：机器型 slug

优点：
- 实现最简单
- 稳定性高

缺点：
- 几乎没有可读性
- 不适合博客公开 URL

结论：采用 **方案 A：pypinyin 转拼音**。

## 4. slug 生成规则

### 4.1 基本规则

统一 slug 生成器遵循以下规则：

1. 英文字母转小写
2. 中文字符转拼音（无声调）
3. 数字保留
4. 标点、空格、特殊字符都视为分隔符
5. 多个分隔符合并为单个 `-`
6. 去掉首尾 `-`
7. 若 slug 冲突，则追加 `-2`、`-3` 等后缀

### 4.2 示例

- `我的第一篇博客` -> `wo-de-di-yi-pian-bo-ke`
- `FastAPI 入门教程 2026！` -> `fastapi-ru-men-jiao-cheng-2026`
- `Python & PostgreSQL 实战` -> `python-postgresql-shi-zhan`

## 5. 稳定性策略

### 5.1 创建时生成，编辑时默认不改

文章 slug 在首次创建时生成。后续编辑文章标题时，默认不自动更新 slug。

原因：
- 避免公开 URL 变化
- 避免历史链接失效
- 更符合博客内容的长期稳定性要求

### 5.2 空结果兜底

若某个输入在清洗后生成空 slug，则回退到：

- `post`
- 若冲突，则 `post-2`、`post-3`

该兜底策略保证极端输入下仍可得到可用 slug。

### 5.3 分类与标签复用同一规则

`Post.slug`、`Category.slug`、`Tag.slug` 应复用统一 slug 生成器，避免同一项目内出现三套不一致的 URL 规则。

## 6. 范围控制

本次升级仅包含以下最小必要范围：

- `app/services/post_service.py`
- `tests/test_post_service.py`
- `requirements.txt`

若后续分类和标签模块开始实现，再直接复用这套规则。

本次不包含：

- 文章创建页面改造
- 分类管理页面
- 标签管理页面
- 历史 slug 重定向机制
- 多语言 slug 策略切换

## 7. 测试设计

本次升级至少补充以下测试：

1. 纯中文标题转拼音
2. 中英混合标题转 slug
3. 标点与特殊符号转分隔符
4. 重复 slug 自动追加序号
5. 极端输入下空 slug 兜底

## 8. 风险与控制

### 8.1 风险

- 引入拼音依赖会增加一个外部库
- 多音字不能保证语义上绝对完美
- 若规则变动过晚，可能影响后续 CRUD 功能一致性

### 8.2 控制措施

- 仅引入轻量拼音依赖
- 本阶段优先保证规则稳定一致，而非追求语言学完美
- 在文章创建功能开始前先升级 slug 规则
- 用自动化测试固化典型中文标题场景

## 9. 结论

项目 slug 规则升级为中文友好方案：

- 英文保留
- 中文转拼音
- 数字保留
- 符号统一作为分隔符
- 重复追加数字后缀
- 空结果回退为 `post`

这是当前中文博客场景下最合适、最稳定、最可维护的 URL 方案。
