# 演进与风险压制守则

日期：2026-03-13

## 目标

本守则用于约束项目后续演进方式，在不引入额外系统复杂度的前提下，持续落实项目核心哲学：

- 轻量
- 克制
- 可扩展

进行较大改动前，建议先配合阅读：

- `docs/engineering/change-review-checklist.md`

本守则关注以下五类高风险方向：

1. 文章模块持续中心化
2. 前后端认证状态不一致
3. 扩展机制平台化倾向
4. 文档驱动实现膨胀
5. 调试产物与临时材料长期堆积

本守则的总原则只有一条：

> 不为消灭局部问题而引入更大的系统复杂度。

---

## 一、文章模块守则

### 1.1 适用范围

以下文件及其后续同类模块属于文章模块核心边界：

- `app/routes/api_v1_posts.py`
- `app/services/post_service.py`
- `app/services/admin_post_service.py`
- `frontend/src/admin/pages/AdminPostsPage.jsx`
- `frontend/src/admin/components/AdminPostEditor.jsx`
- `frontend/src/admin/components/AdminPostList.jsx`
- `frontend/src/admin/components/AdminPostFilters.jsx`

### 1.2 允许继续承载的能力

文章模块允许继续承载以下与博客主闭环直接相关的能力：

- 文章创建、编辑、查询、发布、转草稿
- 文章与分类/标签的直接关联
- Markdown 导入与导出
- 与文章编辑直接相关的最小图片插入能力
- 与文章展示直接相关的预览能力

### 1.3 默认禁止直接加入的内容

以下能力默认不得直接加入文章模块：

- SEO 配置中心
- 文章统计分析
- 历史版本系统
- 审核流或复杂工作流
- 协作编辑
- 自动保存体系
- 大型媒体管理中心
- 与博客最小生产闭环无直接关系的平台能力

如确有必要，必须先回答一句话：

> 这个能力是否属于个人博客最小生产闭环的直接组成部分？

若答案不是明确“是”，默认不做。

### 1.4 结构演进顺序

当文章模块继续增长时，统一按以下顺序处理：

1. 先补边界说明，不急于拆架构
2. 再做一层薄拆分
3. 最后才考虑更明显的结构调整

允许的最小结构调整示例：

- 将 `api_v1_posts.py` 保持为单文件时，按 request model / public read / admin read-write / import-export / publish workflow 分区
- 将 `AdminPostsPage.jsx` 中的状态编排抽成轻量 hook
- 将独立 UI 行为继续下沉到现有子组件

默认不为了“看起来更整洁”而引入：

- 多层 controller / service / usecase / repository 架构
- 全局状态管理方案
- 与当前项目规模不匹配的 CMS 化抽象

### 1.5 改动前自查

凡涉及文章模块新增能力，改动前必须自查：

- 这次改动是不是在把文章模块变成“万能中心”？
- 有没有把非核心博客能力顺手塞进文章模块？
- 能不能继续通过现有边界消化，而不是新增平台层？

## 二、认证状态守则

### 2.1 认证真相定义

本项目认证真相只有一个：

> 后端 session 才是唯一真实认证依据。

以下内容仅属于前端体验层状态，不得视为真实认证：

- `localStorage.myblog_user`
- `hasStoredUser()`
- `AuthGuard`

### 2.2 前端认证边界

前端认证相关模块只承担以下职责：

- 快速决定是否允许进入后台页面壳
- 显示当前用户基础信息
- 在登录/登出后同步本地界面状态

前端模块不得承担：

- 判定 session 是否绝对有效
- 推导后端真实登录态
- 维护复杂认证状态机

### 2.3 统一失配收敛规则

遇到后端返回未授权时，前端应统一执行最小收敛动作：

1. 清理本地用户缓存
2. 跳转到 `/admin/login`
3. 不在多个页面重复实现各自的失效处理逻辑

目标是让错误状态尽快回到一致状态，而不是在前端继续补更多认证逻辑。

### 2.4 默认不新增的认证复杂度

以下能力默认不新增，除非出现明确且持续的真实需求：

- 全局 AuthContext 状态机
- session restore 启动流程
- 复杂登录态预加载
- 多层认证同步机制
- 前端自行缓存更多认证语义

### 2.5 改动判断标准

凡是认证相关改动，先问：

> 这是在减少不一致，还是在为现有轻量认证补一套更复杂的认证框架？

如果更接近后者，默认不做。

## 三、扩展机制守则

### 3.1 当前扩展边界

本项目当前允许的扩展机制边界包括：

- 数据库 provider 扩展
- Hook 事件订阅
- 扩展模块加载
- 前端主题注册

对应位置包括：

- `app/core/database_provider.py`
- `app/core/hook_bus.py`
- `app/core/extension_loader.py`
- `frontend/src/shared/theme/ThemeProvider.jsx`

### 3.2 扩展原则

扩展机制的原则不是“做强”，而是：

> 保持最小可用，并在真实需求出现时做点状增强。

### 3.3 默认禁止的扩展膨胀方向

默认禁止提前建设以下能力：

- 插件元信息体系
- 插件生命周期管理平台
- Hook 优先级、重试或持久化系统
- 热加载插件平台
- 主题包市场
- 复杂主题包系统
- 脱离当前项目需求的通用底层抽象

### 3.4 允许的增强方式

如果未来确有真实扩展需求，只允许按以下顺序演进：

1. 明确一个具体用例
2. 只补这个用例所需的最小能力
3. 保证改动可删、可回退、局部化

例如：

- 需要文章发布后执行一个动作，就补一个 handler
- 不顺手把 Hook 机制升级成通用消息系统

### 3.5 扩展改动检查

每次增强扩展能力前，必须先问：

- 这是为当前真实需求服务，还是为未来假设做准备？
- 这次增强是否会让核心代码被反向污染？
- 如果半年后不需要，这部分代码能否低成本删除？

只要有明显“为未来而设计”的倾向，默认不做。

## 四、文档治理守则

### 4.1 文档与实现关系原则

本项目文档的作用是：

- 说明当前范围
- 记录明确边界
- 帮助开发和理解系统

文档不是自动扩张实现范围的理由。

### 4.2 决策优先级

当文档、实现与需求之间出现张力时，优先级如下：

1. 真实需求
2. 项目哲学（轻量、克制、可扩展）
3. 当前冻结承诺
4. 探索性记录

也就是说：

> 探索性文档不能自动升级为必须兑现的实现义务。

### 4.3 文档分类规则

后续文档内容应尽量分为两类：

#### A. 冻结承诺

包括：

- API 契约
- 当前产品范围
- 当前已确认架构边界
- 工程规范

这类文档应尽量稳定、克制、避免超前承诺。

#### B. 探索记录

包括：

- 规划草案
- 未来方向
- 可能方案
- 讨论性设计

这类文档必须避免写成“已确认能力”。

### 4.4 文档收口原则

发现文档写得比当前合理范围更大时，默认优先选择：

- 修改文档表述
- 删除过度承诺
- 缩小范围描述

而不是默认继续补实现。

### 4.5 文档改动自查

新增或修改文档前，必须自查：

- 这段内容是在描述当前已确认能力，还是在提前许诺未来能力？
- 这份文档会不会让后续开发误以为“必须实现”？
- 这份文档是否超出项目当前产品体量？

## 五、调试产物与人工检查守则

### 5.1 基本原则

所有调试文件、人工检查结果、一次性脚本，默认都视为临时产物。

原则是：

> 任务结束后默认清理，只有明确具有长期价值的内容才保留。

### 5.2 允许长期保留的内容

只有以下内容允许长期保留：

- 阶段性验收材料
- 可重复执行的手工检查脚本
- 明确有复盘价值的检查记录
- 对后续维护有稳定帮助的产物

### 5.3 默认临时内容

以下文件默认属于临时文件，任务结束后应清理：

- `debug_*`
- `todo*`
- `tmp_*`
- `*_result.txt`
- 一次性 HTML 调试页
- 临时截图
- 临时测试数据库
- 一次性浏览器脚本
- 一次性日志文件

### 5.4 命名规范

临时文件必须使用一眼可识别的前缀，优先使用：

- `debug_`
- `todo_`
- `tmp_`

这样便于任务结束时统一清理。

### 5.5 目录收口规则

所有手工检查和调试相关产物应尽量集中存放在约定目录，例如：

- `project_logs/manual_checks/`

不得把一次性调试产物分散在：

- 项目根目录
- `scripts/`
- `frontend/`
- `app/`

等核心代码目录中。

### 5.6 完成标准

任务完成前必须确认：

- 本次启动的进程已关闭
- 临时测试文件已删除
- 一次性调试脚本已清理
- 无无意义结果文件残留
- 保留文件确实具有长期价值

也就是说：

> 不完成清理，不算真正完成任务。

## 六、统一执行检查清单

以后凡是做功能改动、文档更新、调试排查，都建议过一遍下面这份最小清单。

### 6.1 轻量检查

- 有没有引入当前场景不需要的新抽象？
- 有没有为了“以后可能用到”提前设计？
- 有没有把小问题修成大系统？

### 6.2 克制检查

- 这次改动是否真的服务于博客最小闭环？
- 有没有顺手加入不属于当前范围的能力？
- 文档是不是写得比需求更超前？

### 6.3 可扩展检查

- 这次扩展是不是只针对一个真实用例？
- 改动是否局部、可删、可回退？
- 有没有把扩展点做成平台雏形？

### 6.4 工程卫生检查

- 临时文件清理了吗？
- 调试日志清理了吗？
- 进程释放了吗？
- 结果文件是否只保留有长期价值的部分？

---

## 七、一句话版执行原则

如果要把整份守则压缩成几句可执行的话，可以是：

1. 核心模块不许自然长成平台。
2. 前端状态不是真认证，后端 session 才是。
3. 扩展点保持最小可用，不为未来提前建设。
4. 文档优先收口，不用文档逼实现膨胀。
5. 临时产物默认删除，清理是任务完成的一部分。
