# 工程守则与变更检查

日期：2026-03-15

## 目标

本文合并了演进风险压制守则、长期守护规则与变更边界检查清单，为项目后续演进提供统一的风险压制与边界判断入口。

核心原则只有一条：

> 不为消灭局部问题而引入更大的系统复杂度。

---

## 一、变更前检查清单

以下类型改动开始前，建议先过一遍本清单：

- 新功能开发
- 跨模块 Bug 修复
- 架构调整 / 性能优化
- 第三方依赖引入
- 扩展机制增强
- 多文件重构

### 1. 范围边界

- 是否突破 V1 范围（评论、多用户、多角色、审核流、协作编辑等）？
- 是否在把局部需求升级成系统能力（为一个页面引入全局状态体系、为一次导入引入通用工作流平台）？

### 2. 复杂度

- 是否引入了非必要依赖？能否复用现有能力？
- 是否增加运行态复杂度（后台常驻进程、定时任务、缓存层、消息层）？
- 是否破坏默认基线：单体部署 / SQLite / Session 认证 / 无额外基础设施？

### 3. 模块职责

- 是否让已有模块开始承担平台职责？
- 能否继续沿用现有边界消化？（先复用 → 薄拆分 → 最后才考虑新抽象层）
- 热点文件是否开始承载边界外职责？同一规则是否再次多处定义？

### 4. 未来假设

- 是否在为未来假设提前设计？
- 如果需求还没出现，默认不做。

### 5. 文档与测试

- 改动是否同步更新了对应真相文档（product / architecture / api / engineering）？
- 是否需要最小验证（自动化测试或手工验证说明）？

### 快速决策模板

改动前写清三件事：

- **收益**：直接解决什么问题？
- **风险**：会不会让系统变重、变散、变平台化？
- **决策**：直接做 / 缩小做 / 不做 / 改替代方案

---

## 二、文章模块守则

### 允许承载的能力

- 文章 CRUD、发布、转草稿
- 文章与分类/标签的直接关联
- Markdown 导入与导出
- 与文章编辑直接相关的最小图片插入能力
- 与文章展示直接相关的预览能力

### 默认禁止

- SEO 配置中心、文章统计分析、历史版本系统
- 审核流、协作编辑、自动保存体系
- 大型媒体管理中心
- 与博客最小生产闭环无直接关系的平台能力

> 这个能力是否属于个人博客最小生产闭环的直接组成部分？若答案不是明确"是"，默认不做。

### 结构演进顺序

1. 先补边界说明，不急于拆架构
2. 再做一层薄拆分
3. 最后才考虑更明显的结构调整

默认不引入：多层 controller/service/usecase/repository 架构、全局状态管理、CMS 化抽象。

---

## 三、认证状态守则

- 后端 session 才是唯一真实认证依据
- `localStorage.myblog_user`、`hasStoredUser()`、`AuthGuard` 仅属于前端体验层状态
- 遇到后端返回未授权时，统一执行：清理本地缓存 → 跳转 `/admin/login`
- 默认不新增：全局 AuthContext 状态机、session restore、复杂登录态预加载

---

## 四、扩展机制守则

当前允许的扩展边界：

- `app/core/database_provider.py` — 数据库 provider 扩展
- `app/core/hook_bus.py` — Hook 事件订阅
- `app/core/extension_loader.py` — 扩展模块加载
- `frontend/src/shared/theme/ThemeProvider.jsx` — 前端主题注册

原则：保持最小可用，真实需求出现时做点状增强。

默认禁止提前建设：插件元信息体系、插件生命周期管理平台、Hook 优先级/重试/持久化、热加载插件平台、主题包市场。

---

## 五、文档治理守则

- 文档说明当前范围、记录明确边界，不是自动扩张实现范围的理由
- 探索性文档不能自动升级为必须兑现的实现义务
- 发现文档比当前合理范围更大时，优先修改文档、删除过度承诺、缩小范围描述
- 新增文档前先判断现有文档是否可以补充，避免制造平行真相

### 文档优先级

当多份文档出现张力时：

1. `CLAUDE.md`
2. `docs/product/`
3. `docs/architecture/`
4. `docs/api/`
5. `docs/engineering/`
6. `docs/planning/`
7. 历史档案
8. 当前代码实现

## 六、调试产物与工程卫生

### 基本原则

> 任务结束后默认清理，只有明确具有长期价值的内容才保留。

### 默认临时内容（任务结束后清理）

- `debug_*`、`todo*`、`tmp_*`、`*_result.txt`
- 一次性 HTML 调试页、临时截图、临时测试数据库、一次性日志文件

### 目录收口

调试产物集中存放在 `project_logs/manual_checks/`，不得分散在项目根目录、`scripts/`、`frontend/`、`app/` 等核心代码目录中。

### 任务完成标准

- 本次启动的进程已关闭
- 临时测试文件已删除
- 一次性调试脚本已清理
- 保留文件确实具有长期价值

---

## 七、热点文件监控

### 重点对象

- `app/routes/api_v1_posts.py`
- `frontend/src/admin/hooks/useAdminPostsState.js`
- 认证相关 shared 模块
- 扩展点相关 core 模块

### 触发收口条件

出现以下任一情况时，默认应视为进入"需要收口"的状态：

- 热点文件开始承载不属于当前边界的新职责
- 同一规则在多个模块或文档中再次出现平行定义
- 为解决单点问题开始引入更重的抽象层或状态链路
- 相邻改动连续把新增逻辑优先堆进同一个热点文件

一旦触发：先判断边界 → 复用现有结构 → 最小收口 → 最后才考虑结构调整。

---

## 八、SiteSettings 边界

- `SiteSettings` 只承载站点级基础设置与作者基础资料
- 新增字段不直接服务于站点基础信息的，默认不进入 `SiteSettings`
- setup / author 共用同一份站点设置时，优先收口边界说明

## 九、核心回归测试

后续中型及以上改动时，优先确保核心回归测试集合可运行。

详见：[核心回归测试包](./core-regression-suite.md)

---

## 十、长期执行默认判断

1. **默认不做更重的方案** — 除非真实需求已明确到足以支撑复杂度
2. **默认先收口，再扩张** — 能用现有边界消化，就不要急着开新层
3. **默认先验证，再相信直觉** — 涉及主链路、边界、入口、契约的改动，要有最小测试或最小验证

---

## 一句话版执行原则

1. 核心模块不许自然长成平台
2. 前端状态不是真认证，后端 session 才是
3. 扩展点保持最小可用，不为未来提前建设
4. 文档优先收口，不用文档逼实现膨胀
5. 临时产物默认删除，清理是任务完成的一部分
