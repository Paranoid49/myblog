# 轻量扩展机制边界说明

日期：2026-03-12

## 目标

明确项目当前主题系统、扩展加载机制与事件总线（发布订阅）的边界，确保后续扩展继续保持轻量，不滑向插件平台式膨胀。

---

## 一、主题系统边界

当前主题系统位于：

- `frontend/src/shared/theme/ThemeProvider.jsx`

### 当前能力
- 内置主题：`light`、`dark`
- 支持主题持久化到 localStorage
- 支持运行时注册新主题 key：`registerTheme(themeKey)`
- 通过 `data-theme` 写入 `<html>` 控制样式

### 当前边界
主题系统当前只承担：

- 主题 key 切换
- 主题状态持久化
- 主题列表暴露

它**不承担**：

- 复杂主题包加载
- 主题资源动态下载
- 运行时多层 provider 组合
- 插件式主题市场能力

### 约束原则
- 默认只维护 `light` / `dark`
- 只有在真实需求驱动时才新增主题 key
- 不把当前注册机制升级为复杂主题平台

---

## 二、扩展加载机制边界

当前扩展加载位于：

- `app/core/extension_loader.py`

### 当前能力
- 从配置中读取扩展模块路径
- 逐个 `importlib.import_module()` 导入
- 返回成功加载的模块列表

### 当前边界
扩展加载机制当前只承担：

- 最小化模块导入入口
- 启动时执行模块级注册逻辑

它**不承担**：

- 插件生命周期管理
- 插件依赖解析
- 插件安装/卸载系统
- 插件权限或隔离机制

### 约束原则
- 扩展模块应保持最小职责
- 扩展只通过导入完成注册，不引入复杂运行时框架
- 若未来需要更强扩展能力，先确认真实需求，再评估是否升级

---

## 三、事件总线（发布订阅）边界

当前事件总线位于：

- `app/core/hook_bus.py`

### 当前能力
- 订阅事件：`subscribe(event_name, handler)`
- 触发事件：`emit(event_name, payload)`
- 监听器异常隔离，不阻断主流程

### 当前边界
事件总线当前只承担：

- 单进程内的轻量事件分发
- 文章与媒体等关键事件通知

它**不承担**：

- 跨进程事件传输
- 消息持久化
- 重试队列
- 顺序保证
- 分布式消息系统能力

### 约束原则
- 继续作为进程内轻量扩展点使用
- 不把它升级成消息队列替代品
- 若未来出现跨进程需求，再单独评估是否引入专门基础设施

---

## 四、数据库 provider 边界

当前数据库 provider 位于：

- `app/core/database_provider.py`

### 当前能力
- 内置 SQLite / PostgreSQL provider
- 根据 `DATABASE_URL` 解析 driver 并选择 provider

### 当前边界
数据库 provider 当前只承担：

- 数据库 URL 解析
- engine 创建参数差异抽象

它**不承担**：

- 通用 ORM 适配层
- 数据库迁移框架替代
- 多租户数据库编排

### 约束原则
- 继续维持最小 provider 职责
- 新增数据库支持前，先确认是否有真实需求
- 不为假设中的数据库兼容性过度抽象

---

## 五、后续扩展原则

后续如需扩展，统一遵守：

1. 先确认真实需求，而不是为未来假设做平台化设计
2. 优先沿用现有最小入口
3. 不在没有明确收益的情况下增加新的抽象层
4. 优先保持：
   - 核心极简
   - 扩展开放
   - 行为可预测

---

## 六、当前结论

项目当前已经具备：

- 主题扩展入口
- 扩展模块加载入口
- 事件总线（发布订阅）入口
- 数据库 provider 入口

这些能力已经足够支撑当前阶段扩展需求。

下一阶段重点不是“升级扩展系统”，而是：

- 保持这些入口稳定
- 不额外扩张复杂度
- 只在真实需求到来时再做下一步演进
