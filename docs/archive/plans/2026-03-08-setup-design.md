# 首次初始化向导设计

> **历史设计说明（仍有局部参考价值）**
>
> 本文档记录的是初始化向导的设计过程稿。其核心设计结论（如 `/setup`、初始化状态判定、自动登录）大多已落地，
> 但文中部分路由组织、模板命名与早期 SSR 语境已不再代表当前实现。
>
> 因此：
> - 本文档适合用于理解初始化能力为什么这样设计
> - 不应直接作为当前代码结构或当前路由组织的唯一依据
> - 当前实现请以 `app/main.py`、`app/routes/api_v1_setup.py`、`app/services/setup_service.py` 为准

**日期：** 2026-03-08
**主题：** 个人博客首次安装与初始化向导

## 目标

为个人博客提供首次启动即进入 Web 安装向导的能力。未初始化时，用户访问首页 `/` 自动跳转到 `/setup`；安装完成后 `/setup` 永久禁用。安装页负责自动完成数据库结构准备、站点标题持久化、单管理员创建与自动登录。

## 范围

本设计仅覆盖个人博客 MVP 所需能力：

- 首次安装页 `/setup`
- 博客标题持久化
- 单管理员创建
- 初始化状态判断与入口控制
- 初始化完成后的自动登录

明确不做：

- 多用户/多角色系统
- 复杂站点配置中心
- 多站点支持
- 安装完成后的管理员再创建流程

## 方案选择

### 选定方案
使用 **业务数据联合判定初始化状态**：

- 新增 `SiteSettings` 表保存博客标题
- 保留现有 `User` 表承载唯一管理员
- 以“存在 1 条站点配置记录 + 存在管理员用户”作为初始化完成条件

### 未选方案
- **专门初始化状态表**：语义明确，但对当前 MVP 偏重
- **环境变量/本地文件标记**：容易与数据库真实状态不一致，不适合作为初始化依据

## 架构设计

### 数据层
新增 `SiteSettings` 模型与对应 Alembic migration。

建议字段：

- `id`
- `blog_title`
- `created_at`
- `updated_at`

业务约束：

- `site_settings` 在业务上只允许存在一条记录
- 初始化只创建一个管理员账号

### 服务层
新增 `app/services/setup_service.py`，职责限定为：

1. 判断系统是否已初始化
2. 执行首次初始化流程
3. 读取站点配置供模板渲染

不把初始化逻辑塞入 `admin_auth` 或 `main`，避免职责混乱。

### 路由层
新增 `app/routes/setup.py`：

- `GET /setup`：展示安装页
- `POST /setup`：提交安装表单

现有公开页与后台页接入初始化守卫：

- 未初始化时，访问 `/`、`/admin/login`、`/admin/posts*` 统一重定向到 `/setup`
- 已初始化后，访问 `/setup` 重定向到 `/`

### 模板层
新增安装模板 `app/templates/setup/setup.html`。

表单字段：

- 博客标题
- 管理员用户名
- 管理员密码
- 确认密码

### 应用入口
`app/main.py` 仅负责：

- 注册 setup router
- 继续挂载静态资源与既有路由
- 保持入口简洁，不堆叠业务细节

## 初始化状态与请求流转

### 初始化状态定义
系统只有两种状态：

- **未初始化**
- **已初始化**

判定规则：

- 若数据库结构尚未具备所需表，视为未初始化
- 若表已存在但 `SiteSettings` 与管理员未同时存在，视为未初始化
- 只有两者都存在时，才视为已初始化

### 路由行为

#### `GET /setup`
- 未初始化：返回安装页
- 已初始化：302 重定向到 `/`

#### `POST /setup`
- 再次检查是否已初始化
- 若已初始化：拒绝重复安装并跳转 `/`
- 若未初始化：执行 migration + 初始化数据写入 + 自动登录

#### `GET /`
- 未初始化：302 重定向到 `/setup`
- 已初始化：正常展示博客首页

#### 后台相关页
包括：

- `/admin/login`
- `/admin/posts`
- `/admin/posts/new`
- `/admin/posts/{id}/edit`

行为：

- 未初始化：统一跳转 `/setup`
- 已初始化：恢复既有登录/鉴权逻辑

## 数据模型设计

### `SiteSettings`
最小字段集：

- `id`
- `blog_title`
- `created_at`
- `updated_at`

不在本期加入：

- subtitle
- logo
- theme
- SEO 配置
- social links
- analytics

### `User`
保留现有 `User` 模型，不进行结构性重构。

产品语义收窄为：

- 个人博客只允许一个后台管理员
- 初始化完成后不提供新增后台用户能力

## 初始化提交流程

`POST /setup` 推荐顺序：

1. 校验表单
   - `blog_title` 非空
   - `username` 非空
   - `password` 非空
   - `confirm_password == password`
2. 再次检查是否已初始化
3. 执行 Alembic migration，确保数据库结构升级到最新版本
4. 开启数据库事务，写入初始化数据
   - 创建 `SiteSettings`
   - 创建管理员用户
5. 提交成功后写入 session
6. 302 重定向到 `/admin/posts`

## 事务与错误处理

### 一致性原则
- Alembic migration 负责数据库结构准备
- 初始化数据写入必须尽量以单事务完成

### 半初始化语义
若 migration 成功但数据写入失败：

- 系统仍视为未初始化
- 下次访问 `/` 依然回到 `/setup`
- 允许用户重试初始化

### 用户可见错误
允许向用户展示：

- 表单校验失败
- 两次密码不一致
- 用户名冲突
- 初始化失败，请重试

### 用户不可见错误
不直接暴露：

- 原始数据库异常
- Alembic stack trace
- SQL 细节
- 内部密码哈希错误

这些只应记录到服务端日志。

### 重复初始化保护
必须做两层保护：

1. `GET /setup` 已初始化则不展示表单
2. `POST /setup` 提交前再次检查是否已初始化

## 自动登录策略
初始化成功后，直接写入 session 并跳转 `/admin/posts`。

原因：

- 更符合安装完成后的实际操作路径
- 减少一次手动登录步骤
- 提供完整闭环体验

## 标题读取策略
初始化完成后，前台与后台模板显示的站点标题应优先读取 `SiteSettings.blog_title`。

`app/core/config.py` 中的 `settings.app_name` 仅保留为：

- 应用默认名
- 或未初始化状态下的兜底值

## 测试策略

### 路由行为测试
未初始化时：

- `GET /` -> `/setup`
- `GET /admin/login` -> `/setup`
- `GET /admin/posts` -> `/setup`
- `GET /setup` 返回 HTML

初始化成功后：

- `POST /setup` -> `/admin/posts`
- session 中存在 `user_id`
- `GET /setup` -> `/`
- `GET /` 返回博客首页

### 服务层测试
`is_initialized` 覆盖：

- 表不存在
- 只有管理员
- 只有站点配置
- 两者都存在

`initialize_site` 覆盖：

- 成功创建站点配置与管理员
- 博客标题正确保存
- 密码被哈希
- 已初始化后再次执行被拒绝

### 迁移与测试基座
- 生产初始化路径以 Alembic 为准
- pytest 可继续使用轻量数据库初始化方式
- 但需补充 `SiteSettings` 模型与相关 fixture/metadata 验证

## 验收标准

1. 未初始化时访问 `/` 会跳转 `/setup`
2. 未初始化时访问后台相关页面会跳转 `/setup`
3. `/setup` 可提交博客标题、管理员用户名、密码
4. 提交后系统完成数据库结构准备与初始化数据写入
5. 初始化成功后自动登录并跳转 `/admin/posts`
6. 初始化完成后 `/setup` 禁用并重定向到 `/`
7. 首页可读取并显示数据库中的博客标题
8. 系统只允许单管理员初始化闭环，不暴露多用户能力
